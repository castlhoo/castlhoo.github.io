Project

CPTED Crime Risk Prediction with Spark on Kubernetes

2025 Personal Project

Overview

This project is an end-to-end machine learning and MLOps system designed to predict residential crime risk using CPTED (Crime Prevention Through Environmental Design) concepts. I built the workflow around large-scale PySpark modeling and production-style deployment so that the model could move beyond notebook experimentation into an automated and scalable serving pipeline for insurance-related decision support.

Problem Statement

The main goal was to operationalize crime risk prediction in a way that supports real-world usage. Instead of stopping at training a model, I wanted to build a complete system that could process structured crime-related inputs, generate risk predictions, and deploy those predictions through a repeatable CI/CD workflow. In the final design, insurance agents could use the system to estimate residential crime risk and recommend appropriate home insurance products.

Project Walkthrough

1. Technology Stack & System Architecture

The project uses five core technologies: GitHub for source control, PySpark for large-scale model development, Jenkins for CI/CD automation, Docker for containerization, and Kubernetes for scalable deployment and serving.

The full system architecture: a developer pushes code to GitHub, which triggers Jenkins via ngrok webhook. Jenkins builds a Docker image and deploys it to Kubernetes, where the prediction service is exposed to end users — insurance agents and customers — on an Ubuntu-based infrastructure.

2. Data and Modeling

The model uses CPTED-related structured features to predict a Risk Score for each apartment or residential location:

  • Features — Accessibility, Surveillance, Crime Count
  • Target — Risk Score (higher = greater crime vulnerability)
  • Framework — PySpark ML with Random Forest classifier

I created a Spark session, loaded the training CSV, assembled feature vectors, and trained a Random Forest pipeline. The workflow was designed to be reusable for both training and inference at scale, not just a one-off notebook experiment.

3. Webhook and CI Trigger

Since Jenkins was running locally, I used ngrok to expose a public tunnel (https://f06a-182-221-118-26.ngrok-free.app) and configured GitHub Webhooks to send push events to the Jenkins endpoint. This allowed every git push to automatically trigger the full CI/CD pipeline — a key step in making the system behave like a real production workflow.

4. Jenkins Pipeline Design

The Jenkins pipeline was structured with clearly separated stages: Build → Test (DBUnit, JUnit, Jasmine) → Browser Tests (Chrome, Edge, Firefox) → Dev → Staging → Production. Each stage must pass before proceeding, ensuring quality gates at every step of the deployment process.

5. Build and Containerization

After Jenkins detected a code change, it built a Docker image for the PySpark prediction service and pushed it to Docker Hub (castlehoo/crime_prediction_image). This turned the model runtime into a portable artifact that could be consistently deployed in the Kubernetes environment.

6. Docker Build Optimization

I applied two key optimizations to speed up the build process:

  • Multi-stage Docker builds — Separates build dependencies from runtime dependencies, reducing the final image size significantly
  • Docker layer caching — Reuses unchanged build layers (e.g., apt-get install, apt-get update), so only the changed source code layer is rebuilt

7. Kubernetes Deployment and Serving

I stored the Kubernetes access configuration (kubeconfig.yaml) as a Jenkins Secret file credential, following production best practices for secure infrastructure access management.

Jenkins used kubectl with the stored kubeconfig to apply the deployment configuration. The log shows deployment.apps/crime-prediction-app and service/crime-prediction-service successfully applied.

The Kubernetes dashboard confirms the service was created and exposed as crime-prediction-service (NodePort, ClusterIP 10.97.125.66). The model was successfully promoted from a build artifact into a running cluster service.

8. Pipeline Optimization Result

  • Before optimization (Build #5) — Total pipeline time: 2 min 4 sec
  • After optimization (Build #25) — Total pipeline time: 1 min 38 sec

A ~21% reduction in build time through multi-stage builds and layer caching. This matters because repeated CI/CD runs compound quickly, especially with containerized ML systems.

What I Gained

  • End-to-end MLOps pipeline construction — Built the full path from model training to production serving (PySpark → Docker → Jenkins → Kubernetes), understanding how each component connects in a real deployment workflow.
  • CI/CD automation for ML systems — Learned to set up webhook-triggered pipelines where a single git push automatically builds, tests, containerizes, and deploys a model to a running cluster.
  • Docker optimization for production — Applied multi-stage builds and layer caching to reduce image size and build time, skills that directly transfer to any containerized production system.
  • Kubernetes deployment and service exposure — Gained hands-on experience deploying ML services to Kubernetes, managing credentials securely, and exposing prediction endpoints as cluster services.
  • Bridging ML and engineering — This project reinforced that building a good model is only half the job; the other half is making it deployable, reproducible, and maintainable in production.
