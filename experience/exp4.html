Project Walkthrough

1. Infrastructure Architecture Overview

The project was an AWS migration for Woori Financial Securities. I designed the overall infrastructure architecture with three core design principles:

  • Security — Bastion Host for secure access, private subnet isolation, and SSH-based access control
  • Manageability — EKS with 4 node groups and 6 nodes, centralized operations, and auto recovery (health check + scaling + rescheduling)
  • High Availability — Multi-AZ deployment with fault tolerance across zones and continuous service during failures

2. Overall System Architecture

The system was built on three pillars:

  • Terraform (IaC) — Version-controlled and reproducible infrastructure with batch deployment of AWS services, reducing manual errors
  • High Availability (Kubernetes) — Redis cluster + multi-pod deployment with no single point of failure, load balancing across pods, and auto recovery
  • Data Separation — Sensitive and general data isolated for improved security, regulatory compliance, and optimized performance

3. Real-time Market Data Pipeline

I built an end-to-end real-time data delivery pipeline for market data: Korea Investment API → Redis Pub/Sub → WebSocket.

  • Data Collection (Publisher) — Standardizes external API data, handles errors and reconnection, controls API load with rate limiting
  • Data Distribution (Redis Pub/Sub) — In-memory for ultra-low latency, one-to-many real-time broadcasting with high scalability
  • User Delivery (WebSocket) — Real-time push with ms-level latency, persistent connection with no HTTP overhead

For persistent storage, I designed a batch storage pipeline: Redis Cache → Scheduled Batch → MySQL. This reduced DB load, enabled faster real-time queries via cache, and used efficient bulk INSERT for optimized transactions.

4. Database Replication

I implemented read/write separation using Master (write) + Read Replica (read) for load distribution and scalability.

  • High Availability — Standby DB for failover, minimizing downtime and data loss
  • AOP Routing — Read queries route to Replica, write queries route to Master, keeping the logic transparent and maintainable

5. CI/CD Pipeline

I built an automated CI/CD pipeline: Developer → GitHub → Bastion Host → Jenkins → Lambda → ECR → ArgoCD.

  • Secure Integration — GitHub → Jenkins via Bastion Host with IP Whitelist for webhook security
  • Automation — Git push triggers auto build and deploy; API Gateway + Lambda for serverless triggers
  • Image Management — Docker → ECR (private registry) with IAM and encryption
  • GitOps (ArgoCD) — Auto sync from Git to EKS with easy monitoring and rollback
  • Deployment Strategy — Backend and Frontend separated for independent scaling and deployment

6. Service Flow & User Experience

The end-to-end service flow was designed as a 3-layer architecture:

  • User Access (Frontend) — WAF → Route53 → IGW → ALB for secure entry and traffic routing
  • API Processing (Backend) — Internal ALB → Backend services with secure internal communication and load balancing
  • Data Layer — Master/Replica DB with read/write separation + Redis cache for fast response

7. EKS Cluster Configuration

I configured the EKS cluster with 4 dedicated node groups:

  • Frontend (React) — 2 nodes
  • Backend (Spring) — 2 nodes
  • Monitoring (Prometheus, Grafana) — 1 node
  • Redis (Cache, Pub/Sub) — 1 node

8. Frontend & Backend Architecture

  • Frontend — Internet-Facing ALB as the entry point for external traffic with SSL termination (HTTPS), routing traffic to React nodes only via Taint and Tolerance
  • Backend — Internal ALB handling internal API traffic only, routing to Spring nodes
  • ALB Strategy (3-Tier) — Frontend ALB for external traffic, Backend ALB for internal traffic, cleanly separated

9. Auto-Scaling (HPA)

I configured Horizontal Pod Autoscaler (HPA) to handle dynamic load:

  • Scale based on 70% memory usage threshold
  • Min 2 pods / Max 5 pods
  • High load → auto scale out, normal load → scale in
  • Benefits: elastic scaling, cost optimization, and stable performance

10. Testing & Validation

Test Scenario 1: IPO Access Rush Response — Simulated a traffic spike scenario (IPO rush). Standard setup: 70% memory threshold with 2-5 pods. Under IPO surge, pods auto-scaled out within ~30 seconds, handling traffic spikes stably.

Test Scenario 2: RDS Failover — Tested Multi-AZ failover with Master + Standby configuration. When the primary AZ (2a) failed, the system auto-detected the failure and promoted Standby to Master. Result: switch time under 2 minutes, 0% data loss (RPO = 0), and minimal downtime (RTO < 2 min).
